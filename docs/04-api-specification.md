## 1. Scope and normative terminology

This document is the human-readable contract for the complete commercial REST API. The machine-readable contract shall be OpenAPI 3.1 and must preserve these semantics. Paths are relative to:

`https://api.example.com/api/v1`

### 1.1 Mandatory terminology mapping

| Requested or ambiguous term | Canonical public API term | Contract rule |
|---|---|---|
| Organization / SaaS tenant | **Organization** | The SaaS customer, subscription, configuration, data, and security boundary. Every ordinary access JWT carries exactly one immutable `org_id` for its lifetime. |
| Tenant meaning a renter | **Resident** | Renters are exposed only through `/residents` or self-scoped `/resident/*`. **No `/tenants` endpoint exists**, because tenant is reserved for internal SaaS isolation language. |
| Contract | **Lease** | Rental contracts are exposed as `/leases`; no `/contracts` alias exists. |
| Room | **Unit** | Apartments, studios, private rooms, and shared rooms are Unit types. Units are exposed through `/units`; **no `/rooms` endpoint or ROOM table exists**. |
| Building | **Building** | An optional physical grouping beneath a Property. A Unit may reference a Building but does not require one. |
| Electric meter / water meter | **Meter** | One `/meters` resource uses `meterType=ELECTRICITY\|WATER`; duplicate electricity- or water-specific APIs are forbidden. |
| Property owner | **Property Owner** | A person or legal-entity business Party. Property Owners are distinct from Organization Owners and Users; ownership or a Management Agreement never grants application access. |

The canonical rentable hierarchy is `Property > Unit > optional Bed`. A Bed is available only for named bed-level assignment in a shared Unit; capacity-managed shared Units require no Bed records.

## 2. Protocol, representation, and naming rules

### 2.1 HTTP, media types, and methods

- TLS is mandatory. JSON request and response media type is `application/json`; errors use `application/problem+json`.
- JSON field names are `camelCase`; path segments are lowercase plural nouns and multiword segments use kebab-case.
- `GET` and `DELETE` have no request body. `POST` creates resources or executes explicit workflows. `PATCH` applies a documented partial merge shape; omitted fields are unchanged and explicit `null` is allowed only where the field is nullable.
- UUIDs are canonical lowercase RFC 9562 UUID strings and are opaque. Clients must not infer ordering or tenancy from IDs.
- Date-only values use `YYYY-MM-DD`. Instants use RFC 3339 with an offset, normally UTC `Z`. Business schedules retain an IANA `timeZone`.
- Effective ranges are start-inclusive and end-exclusive. A missing `effectiveTo` means open-ended.
- Enums are uppercase stable tokens. Additive enum values are backward-compatible; clients must preserve safe unknown-value behavior.
- Boolean values are JSON booleans, never `0`/`1` strings. Authoritative numeric quantities use decimal strings where fractional precision matters.
- Money is `Money { amount: decimal-string, currency: ISO-4217 }`. Binary floating-point values are forbidden. Cross-currency totals require an explicit rate source and timestamp.

### 2.2 Common headers

| Header | Direction | Required behavior |
|---|---|---|
| `Authorization: Bearer <access-token>` | Request | Required except where an endpoint is explicitly Public. |
| `Content-Type: application/json` | Request | Required when a JSON body is present. |
| `Accept: application/json` | Request | Recommended; unsupported media negotiation returns `406 NOT_ACCEPTABLE`. |
| `Idempotency-Key` | Request | Required on rows marked **Idempotency required**; 1–128 printable ASCII characters. |
| `If-Match` | Request | Required on rows marked **If-Match required**. Value is the exact ETag from the latest representation. |
| `If-None-Match` | Request | Optional on cacheable `GET`; a match returns `304`. |
| `X-Request-ID` | Both | Optional caller correlation ID; server returns a validated value or generates one. |
| `Traceparent` | Both | Optional W3C trace context; never used for authorization. |
| `ETag` | Response | Returned for versioned resources. |
| `Location` | Response | Returned for created resources and accepted operations. |
| `Retry-After` | Response | Returned for throttling, temporary dependency failure, and recommended operation polling. |
| `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset`, `RateLimit-Policy` | Response | Describe the effective rate-limit bucket where available. |
| `Deprecation`, `Sunset`, `Link` | Response | Describe deprecated operations and migration documentation. |

`X-Tenant-ID`, `X-Organization-ID`, and equivalent organization-selection headers are unsupported and return `400 ORGANIZATION_HEADER_FORBIDDEN`.

## 3. Standard representations

### 3.1 Resource and collection envelopes

`ResourceEnvelope<T>` contains `data: T` and optional `meta`. Every mutable resource includes `id`, useful `organizationId`, `status`, `version`, `createdAt`, and `updatedAt`; `organizationId` is informational and never authorization evidence. Responses may also include `links.self`.

`CollectionEnvelope<T>` contains:

| Field | Type | Meaning |
|---|---|---|
| `data` | `T[]` | Authorized, ordered items. |
| `page.limit` | integer | Applied page size. |
| `page.hasNext`, `page.hasPrevious` | boolean | Navigation availability. |
| `page.nextCursor`, `page.previousCursor` | string or null | Opaque cursor tokens. |
| `meta.asOf` | timestamp, optional | Snapshot/read-model instant where applicable. |
| `meta.freshnessSeconds` | integer, optional | Projection lag where applicable. |

Create responses return `201` and the created `ResourceEnvelope<T>`. Successful updates return `200` and the updated envelope unless the endpoint explicitly returns `204`.

### 3.2 Async operation resource

`Operation` fields are `id`, `type`, `status=QUEUED\|RUNNING\|SUCCEEDED\|PARTIALLY_SUCCEEDED\|FAILED\|CANCELLED`, `progress.completed`, `progress.total`, `progress.percent`, `result.resourceLinks[]`, optional `result.downloadUrl`, optional `result.expiresAt`, bounded `errors[]`, `createdAt`, `startedAt`, `completedAt`, and `links.self`. Async submission returns `202`, `Location: /api/v1/operations/{operationId}`, and `ResourceEnvelope<Operation>`.

Large error sets are retrieved from an authorization-checked short-lived error document. Cancellation is best effort: accepted cancellation returns `202`; an already terminal or non-cancellable operation returns `409 OPERATION_NOT_CANCELLABLE`.

## 4. Collection queries

`CollectionQ` means: `limit` default `25`, minimum `1`, maximum `100`; exactly one of `after` or `before`; `sort` from the endpoint allowlist; optional `fields` and `include` from endpoint allowlists. Cursors are opaque, integrity-protected, expire after at least 24 hours, and bind normalized filters, sort, scope, and direction. Invalid or mismatched cursors return `400 CURSOR_INVALID`.

Search parameter `q` is trimmed text, minimum 2 and maximum 100 characters, over endpoint-documented fields. Filters use explicit typed parameters; comma-separated enum/UUID lists contain at most 50 values. Date ranges use `field[gte]`, `field[gt]`, `field[lte]`, and `field[lt]`. Sort syntax is `sort=field,-otherField`; the server appends `id` as a deterministic tie-breaker. Unsupported filters, fields, includes, or sorts return `400 QUERY_PARAMETER_INVALID`.

No total count is returned by default. Endpoints that support it accept `count=none\|approximate` with default `none`. Availability and reporting endpoints expose `asOf`; mutable sort keys may still yield duplicates/skips across pages unless a documented snapshot is used.

## 6. Organization isolation and authorization

Every ordinary access JWT has exactly one `org_id`, one `membership_id`, a human/service subject, session ID where applicable, audience, assurance claims, and expiry. Services verify issuer, audience, signature, time, session, token version, active membership, exact permission, property scope, resource relationship, lifecycle, entitlements, separation of duties, and step-up requirements.

- A protected route containing `{organizationId}` requires it to equal token `org_id`; mismatch returns `404 RESOURCE_NOT_FOUND` before lookup. The sole exception is token exchange.
- `organizationId` in a representation is informational. Request body/query/path values never select the write partition or prove access.
- Out-of-Organization, out-of-property, and cross-resident objects normally return `404`; an in-scope object with insufficient action permission returns `403`.
- Resident self-scope is derived from the authenticated identity-to-Resident and Lease relationships. Client-supplied resident, lease, property, unit, or household IDs never broaden it.
- Background operations snapshot the initiating Organization, actor, permission, property scope, and approved filters; workers revalidate scope before reading or writing.
- Property ownership and Management Agreements are business facts, not authorization grants.
- Platform administration uses separate routes and token audience and has no implicit Organization-data access.

## 7. Idempotency, concurrency, status, and lifecycle conventions

### 7.1 Idempotency

Rows marked **Idempotency required** require `Idempotency-Key`. The key is scoped by `(org_id, actor identity, normalized HTTP method and route)`. The server stores a canonical request hash, processing owner/state, and original response. The default TTL is 24 hours; financial mutations, payment callbacks, invoice posting, refunds, and deposit actions retain deduplication for at least 7 days. An equivalent retry returns the original status/body and `Idempotency-Replayed: true`; a changed request returns `409 IDEMPOTENCY_KEY_REUSED`. Expired deduplication metadata never removes the resulting business record.

### 7.2 Optimistic concurrency

Mutable resources expose an `ETag`. Rows marked **If-Match required** reject an absent precondition with `428 PRECONDITION_REQUIRED` and stale values with `412 VERSION_MISMATCH`. A matching ETag does not guarantee a business action: inventory, money, and workflow checks can still return `409`.

### 7.3 Standard statuses

`200` read/update/action with representation; `201` resource created; `202` async accepted; `204` successful action/deactivation without body; `207` explicitly documented small synchronous bulk result; `304` conditional read; `400` malformed/query error; `401` unauthenticated; `403` known in-scope denial; `404` absent/concealed; `409` business conflict; `410` permanently expired token; `412` stale version; `413` too large; `415` wrong media type; `422` semantic validation; `423` locked; `428` missing precondition; `429` throttled; `5xx` safe server/dependency failure.

Archive/retire/end/action endpoints preserve financial, legal, occupancy, and audit history. Ordinary hard deletion is not exposed for such records.

### 7.4 Bulk operations

Bulk requests contain `items[]` with unique `clientItemId`, plus `mode=PARTIAL\|ATOMIC` where supported. Default maximums are 1,000 lightweight rows, 100 ordinary mutations, and 25 financially sensitive items; endpoint-specific lower limits prevail. The normal result is `202 Operation`. A documented synchronous batch may return `207` with one `SUCCEEDED\|FAILED\|SKIPPED` result per item. Batch-level authentication, schema, scope, and size failures reject the whole request; item-level domain failures use stable codes.

## 8. Rate limits, caching, versioning, deprecation, and webhooks

Layered buckets apply by edge/IP, account/session, Organization, service credential, and risk-sensitive route. Authentication, search, signed URLs, financial mutations, reports/exports, imports, and provider callbacks have separate policies. Default expensive-work concurrency is two queued/running reports or exports per Organization. A `429` includes `Retry-After` and rate-limit headers. Valid retries still count toward abuse controls.

Authenticated data is `Cache-Control: private` or `no-store`; authentication and signed-URL responses are `no-store`. Stable reads may use `ETag`/`If-None-Match`. Dashboard and report projections state `asOf` and freshness.

The major version is in `/api/v1`. Adding endpoints, optional fields, filters, events, or safe enum values is backward-compatible. Removing/renaming fields, changing types/semantics, strengthening optional request requirements, or weakening isolation requires a new major version. Deprecation uses `Deprecation`, `Sunset`, and `Link: <url>; rel="deprecation"` with at least 12 months' notice and at least 6 months after a GA replacement, except documented security/legal/provider emergencies.

Outbound webhooks use an envelope containing `id`, `type`, `apiVersion`, `schemaVersion`, `organizationId`, `occurredAt`, and bounded `data.resourceType`, `data.resourceId`, `data.status`. Delivery is at least once. The service signs `<unixTimestamp>.<rawBody>` with HMAC-SHA256 and sends `Webhook-Id`, `Webhook-Timestamp`, and `Webhook-Signature: v1=<lowercase-hex>`. Receivers should enforce a five-minute timestamp window and deduplicate event IDs. Retries use exponential backoff with jitter for network errors, `408`, `425`, `429`, and `5xx`. Ordering is guaranteed only where an event contract explicitly states per-aggregate ordering.

## 23. Documents and private files

Object storage is private S3-compatible storage. The API never returns permanent public URLs. Upload URLs constrain bucket/key, method, exact or bounded length, MIME allowlist, checksum, and expiry. Default maximum file size is 25 MiB, configurable by category/plan; the upload-intent response states the effective limit. Default upload URL TTL is 15 minutes and download URL TTL is 5 minutes.

Completion verifies ownership, stored size, detected MIME, and SHA-256 checksum, then moves the version to `SCANNING`. A version cannot be linked into an executing workflow, previewed, or downloaded until `READY`. Malware, archive file count/expanded size/nesting/compression ratio, scan timeout, retention, and legal hold are enforced.

## 27. Contract governance and OpenAPI requirements

The OpenAPI 3.1 artifact generated from this contract shall:

1. Define every endpoint row, named request shape, response envelope, permission requirement, parameter limit/default, header requirement, and stable problem code in this document.
2. Use reusable schemas without weakening required fields or domain semantics. `oneOf`/discriminators shall model typed scopes, payment channels, Meter scopes, notification audiences, and async results.
3. Declare security per operation. Public operations shall explicitly use an empty security requirement; protected operations shall declare the Organization-scoped bearer scheme and document self/property/Organization scope.
4. Declare `Idempotency-Key` and `If-Match` as required headers exactly where marked and cover replay, hash conflict, stale version, and missing precondition in contract tests.
5. Publish enum, decimal-string, UUID, date, timestamp, IANA timezone, ISO-4217, checksum, cursor, and RFC 9457 schemas.
6. Keep examples fictional, bounded, and free of real personal, financial, credential, provider, or storage data.
7. Run isolation tests for wrong Organization, wrong property, wrong resident, nested-parent mismatch, service audience mismatch, and attempted Organization selection through headers/body/query.
8. Run workflow/concurrency tests for inventory overbooking, billing/posting replay, Payment callback replay, allocation/refund races, meter rollback/reset/correction, document scan gates, audience resolution, and operation cancellation.
9. Review every API-affecting change with product, domain, security, and client owners. SDKs and changelogs are versioned from the published OpenAPI artifact.

No endpoint in this contract implies statutory accounting, tax filing, legal compliance, payment-card custody, owner distributions, or unrestricted platform/support access.
